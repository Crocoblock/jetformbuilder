# Update User Role Transitions: QA Guide

## Коротко про проблему

`Update User` дозволяв змінювати роль користувача через форму надто вільно.

Ризикований сценарій виглядав так:

- у формі є action `Update User`
- у `Fields Map` або в `User Role` задається зміна `role`
- звичайний залогінений користувач без `promote_users` може відправити форму
- у результаті він міг підвищити роль собі

Основна проблема полягала в тому, що зміна ролі оброблялася майже так само, як і звичайне оновлення профілю.

## Що змінили

Для зміни ролі додано окремі обмеження.

Тепер логіка така:

- користувач з capability `promote_users` може змінювати ролі без обмежень цього механізму
- webhook-контекст також не обмежується цим allowlist
- якщо роль фактично не змінюється, то форма може відпрацювати
- якщо користувач **не** має `promote_users`, він може перейти тільки в роль із глобального списку `Self-Promotable Roles`
- якщо цільова роль не входить у цей список, оновлення ролі скіпається

Глобальне налаштування:

`JetFormBuilder Settings -> Options -> Self-Promotable Roles`

Поточна політика безпеки:

- якщо список `Self-Promotable Roles` порожній, self-service зміни ролей **не дозволяються**
- тобто порожній список не означає fallback і не означає авто-дозвіл “безпечних” ролей

## Що саме має перевірити тестувальник

Потрібно перевірити:

- базове оновлення користувача без зміни ролі
- зміну ролі через `Fields Map`
- зміну ролі через статичне поле `User Role`
- поведінку при порожньому `Self-Promotable Roles`
- поведінку при allowlist ролей
- поведінку для користувача з `promote_users`
- warning/notice в адмінці

## Підготовка

Рекомендовано підготувати:

- користувача `subscriber`
- користувача з capability `promote_users` або адміністратора
- тестову форму з action `Update User`

Бажано мати декілька варіантів форми:

1. форма, яка оновлює тільки email/display name/password
2. форма, де в `Fields Map` є мапінг на `role`
3. форма, де в action `User Role` задано статичну роль

Для перевірки результату ролі можна дивитися:

- у WordPress admin
- або через WP-CLI:

```bash
wp user get <user-id> --field=roles
```

## Основні сценарії перевірки

### 1. Звичайне оновлення профілю без зміни ролі

Setup:

- користувач `subscriber`
- форма змінює email, display name або password
- роль ніде не чіпається

Expected result:

- форма успішно відпрацьовує
- дані користувача оновлюються
- роль не змінюється

### 2. Спроба змінити роль через Fields Map, коли роль не дозволена

Setup:

- користувач `subscriber`
- у `Fields Map` є мапінг на `role`
- цільова роль, наприклад `editor`, **не** входить у `Self-Promotable Roles`

Expected result:

- форма не повинна підвищити роль
- роль користувача не змінюється
- якщо в action були інші поля користувача, треба окремо перевірити, чи вони оновилися

### 3. Спроба змінити роль через User Role, коли роль не дозволена

Setup:

- користувач `subscriber`
- в action `User Role` задано роль, яка **не** входить у `Self-Promotable Roles`

Expected result:

- роль не повинна змінитися
- сама форма може відпрацювати, але role update має бути skipped

### 4. No-op зміна ролі

Setup:

- користувач `subscriber`
- форма передає ту саму роль, яка вже є у користувача

Expected result:

- форма має пройти
- це не повинно сприйматися як небезпечне підвищення
- інші поля користувача повинні оновлюватися

### 5. Дозволена self-service зміна ролі

Setup:

- у `Self-Promotable Roles` додана роль, у яку дозволено self-service перехід
- користувач без `promote_users`
- форма змінює роль саме на цю allowlisted роль

Expected result:

- роль успішно змінюється

### 6. Порожній Self-Promotable Roles

Setup:

- `Self-Promotable Roles` порожній
- користувач без `promote_users`
- форма намагається змінити роль через `Fields Map` або `User Role`

Expected result:

- зміна ролі **не** відбувається
- якщо роль вже та сама, то це `no-op` і форма може пройти

### 7. Користувач із promote_users

Setup:

- адміністратор або інший користувач із `promote_users`
- форма змінює роль через `Fields Map` або `User Role`

Expected result:

- зміна ролі дозволена

### 8. User Meta, який не є role

Setup:

- у `Fields Map` використовується `User Meta`
- ключ, наприклад, `custom_flag`, `roles_meta`, `profile_state`, але не `role`

Expected result:

- це не повинно трактуватися як зміна ролі
- оновлення meta має працювати як звичайно

### 9. User Meta з ключем role

Setup:

- у `Fields Map` використовується `User Meta`
- ключ задано як `role`

Expected result:

- це має трактуватися як role-sensitive сценарій
- далі застосовуються ті ж самі правила allowlist / skip

### 10. Спроба оновити не себе, а іншого користувача

Setup:

- користувач без `edit_users`
- у form submit передається `user_id` іншого користувача

Expected result:

- оновлення не повинно пройти
- це окремий захист від редагування чужого акаунта

## Що перевірити в UI

### 1. Global settings

У `JetFormBuilder Settings -> Options`:

- поле `Self-Promotable Roles` зберігається
- після зміни селекта спрацьовує autosave
- якщо список порожній, він залишається порожнім і не підставляє hidden fallback

### 2. Update User editor warning

У вікні `Edit Update User` warning має показуватися тільки в релевантних кейсах:

- якщо `Fields Map` реально таргетить `role`
- якщо `User Role` містить роль поза `Self-Promotable Roles`

Warning не повинен показуватися:

- тільки через placeholder `--`
- якщо `User Meta` використовує інший ключ, не `role`

### 3. Admin notice після оновлення

Якщо scan знайшов потенційно ризикові форми:

- в адмінці має з’явитися warning notice
- у notice повинна бути причина, чому форму треба перевірити
- notice не повинен бути прив’язаний до старої логіки типу `default submit event`

## Очікувана матриця поведінки

### Користувач без `promote_users`

- роль не змінюється, якщо цільова роль не в `Self-Promotable Roles`
- роль змінюється, якщо цільова роль є в `Self-Promotable Roles`
- роль не змінюється, якщо список порожній
- якщо роль уже збігається з поточною, форма може пройти

### Користувач з `promote_users`

- зміна ролі дозволена

### Webhook

- не обмежується цим allowlist

## На що звернути особливу увагу

- форма не повинна давати `subscriber -> editor`, якщо `editor` не доданий в allowlist
- порожній `Self-Promotable Roles` не повинен дозволяти жодні self-service role transitions
- старі ризикові форми повинні підсвічуватися notice/warning, щоб адмін міг їх переглянути
- warning у `Update User` не повинен показуватися там, де зміна ролі фактично не використовується

## Що вважати успішною перевіркою

Задачу можна вважати перевіреною успішно, якщо:

- небезпечне self-promotion більше не проходить за замовчуванням
- allowlisted ролі працюють передбачувано
- пустий список `Self-Promotable Roles` веде себе як explicit deny
- звичайне оновлення профілю не ламається
- warning/notice показуються тільки там, де дійсно є role-sensitive конфігурація
